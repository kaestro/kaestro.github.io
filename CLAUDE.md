# kaestro.github.io 프로젝트 가이드

## 블로그 글 작성 가이드라인

### Front Matter 필수 항목

```yaml
---
layout: default
classes: wide
title: "제목"
subtitle: "부제목"
description: "SEO용 요약 (1~2문장, 핵심 키워드 포함)"
date: YYYY-MM-DD
categories: 개발이야기  # 또는 다른 카테고리
---
```

- `description`: 검색엔진 노출용. 핵심 기술 키워드를 포함하되 자연스러운 문장으로 작성

### 글 톤 & 스타일 (개발이야기 카테고리)

기술 블로그지만 **읽히는 글**이어야 한다. 논문이 아니라 에세이에 가깝게.

1. **감정을 드러낼 것**: "당혹감", "자존심에 스크래치", "화끈거림" 같은 솔직한 감정이 공감을 만든다
2. **은유와 비유를 적극 활용**: 기술적 문제를 "빌런", "시한폭탄", "기도하는 배포" 같은 비유로 풀어 비개발자도 맥락을 이해할 수 있게
3. **짧고 펀치력 있는 문장**: 한 문장이 두 줄을 넘기지 않도록. 긴 설명보다 임팩트
4. **대화체 삽입**: "오늘 안에 못 끝내겠죠?", "이제 될 수도 있을 것 같아요" 같은 실제 대화가 현장감을 살림
5. **Before/After 비교표**: 결과 섹션에서 시각적 대비 효과. 독자가 한눈에 변화를 체감
6. **코드 블록은 전문성의 증거**: 개념 수준이 아닌, 실제 설정 파일이나 명령어를 보여줘 기술 블로그로서의 신뢰 확보
7. **이미지 삽입**: 테스트 통과 스크린샷, CI 대시보드 등 실제 결과물을 시각적으로 보여줄 것

### 피해야 할 것

- **건조한 기술 보고서 톤**: "~를 수행했다", "~로 변경했다"만 나열하면 읽히지 않음
- **내부 이슈 번호 직접 노출**: MF-902 같은 이슈 번호는 글에 넣지 않음. 서술의 근거로만 활용
- **과도한 겸손**: "별것 아니지만~" 대신 자신이 한 일에 대한 자부심을 드러낼 것

### 이미지 경로 규칙

```
/images/<글 제목 또는 주제>/<파일명>.png
```

마크다운에서 참조 시 공백은 `%20`으로 인코딩:
```markdown
![설명](/images/스테이징%20환경%20정비기/ci-validation-pass.png)
```

```
│   └── ...

```
